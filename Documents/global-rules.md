## 2. 文档体系规范

### 2.1 三层记忆结构 [2026-07-20]

| 层级 | 文件 | 定位 | 更新时机 |
|------|------|------|---------|
| L1 | `Documents/global-rules.md` | 跨项目通用规范 | 新习惯/规则形成时 |
| L2 | `{project}/Documents/{project}-memory.md` | 单个项目决策记忆 | 合并 main 时统一更新 |
| L3 | `{project}/Documents/{project}-context.md` | 策划文档 + 技术速查 | 合并 main 时统一更新 |

### 2.2 上下文文档同步规则 [2026-07-16]

- 同步时机：仅在更新 main 分支或合并分支时向用户确认是否需要同步上下文文档
- 功能分支上日常开发修改时不需要确认
- 同步时必须进行全覆盖检查，不能只更新与当天修改相关的几个点：
  - 操作键/按键表
  - 核心属性/玩家属性
  - 渲染常量（确认值是当前实际值）
  - 架构概览（确认新增系统模块已加入）
  - TILE系统/楼层系统等核心系统描述
  - 章节编号（无重复无遗漏）
  - 函数索引（补充所有新增函数）
  - 数据流（补充启动/探索/楼层等完整流程）
  - 游戏流程（补充探索模式+战斗模式完整流程）
  - UI面板（补充楼层信息栏等新增UI）
  - 文件清单（补充新配置文件）
  - 下一步方向（移除已完成的）
  - 过期内容清理（已淘汰的描述）

### 2.3 合并 main 优化流程 [2026-07-16 优化]

当用户说"同步项目文档，提交并推送，并合并到main分支"时：

1. **同步文档**：先更新对应的上下文文档（context.md + memory.md），在"最近更新记录"顶部新增一条
2. **全面检查变更**：`git status` + `git diff` 仔细检查所有变更文件
3. **一次性提交**：所有相关变更（代码 + 文档）纳入同一个 commit
4. **推送到远程**：push 当前分支到 origin
5. **合并到 main**：切到 main → pull → merge 当前分支 → push main
6. **切回开发分支**：回到当前功能分支

**关键优化**：先更新文档再提交，避免之前"先提交代码 → 合并main → 再提交文档 → 再合并main"的重复操作。

### 2.4 多端开发记忆同步 [2026-07-20]

- **切换电脑前**：`commit + push` 当前分支（即使不合并 main），确保三份记忆文件已更新
- **切换电脑后**：`pull` 最新，然后主动告知 AI 读取三份文档
- **新会话启动**：主动提示 AI 读取 `global-rules.md` + 当前项目的 `{project}-memory.md` + `{project}-context.md`

## 4. AI 会话规范

### 4.1 新会话启动约定 [2026-07-20]

- AI 会自动加载 CodeBuddy Memories 中的精简规则
- 用户主动告诉 AI 读取三份文档（global-rules + memory + context）
- 三文件通过 Git 多端同步，保证不同电脑间记忆一致

### 4.2 记忆更新原则 [2026-07-20]

- **L2 memory.md** 记录标准："如果在另一台电脑不知道这件事，会不会做出错误决策或重复踩坑？" → 会就记
- 每条决策带时间戳、原因、被否决方案
- 反复变更（X→Y→X）要保留完整演变路径
- 不确定是否该记时，主动询问用户

### 4.3 不可回溯的边界 [2026-07-20]

- 如果在某台电脑上的对话没有被及时整理成 memory.md 并 push，切换电脑后无法回溯
- 解决方案：严格执行第 2.4 节的切换前 commit+push 流程
