## 1. Git 使用规范

### 1.1 Monorepo 分支策略 [2026-07-16]

| 分支 | 用途 |
|------|------|
| `main` | 稳定主线，只合并经过验证的版本 |
| `dev/match3` | match3-rpg-project 专属开发分支 |
| `dev/tarot` | tarot-battle-project 专属开发分支 |

- 每次打开项目或新会话时，根据当前工作项目自动切换到对应分支
- 无法判断当前项目时，主动询问用户选择

### 1.2 main 分支保护 [2026-07-16]

- 直接对 main 分支进行修改前，必须先警告提醒用户当前操作会直接修改 main 分支
- 询问用户确认：要继续直接改 main，还是改为走功能分支流程
- 用户明确确认后才能继续

### 1.4 AI Git 写入操作权限边界 [2026-07-20]

**核心原则：AI 绝不得在用户未明确要求时自动执行任何 Git 写入操作。**

受限制的操作（禁止自动执行）：
- `git commit` — 创建提交
- `git push` — 推送到远程
- `git merge` — 合并分支
- `git rebase` — 变基操作
- `git reset --hard` / `git checkout --` 等破坏性操作
- 任何会修改 Git 历史或远程仓库的操作

允许的操作（无需用户确认）：
- `git status` — 查看状态
- `git diff` — 查看差异
- `git log` — 查看历史
- `git branch` — 查看分支
- 其他纯读取操作

**触发条件**：只有当用户当前消息中包含以下任一明确指令时，AI 才能执行 Git 写入操作：
- "提交" / "commit"
- "推送" / "push"
- "合并" / "merge"
- "同步文档"（该指令隐含后续的 commit+push+merge 流程）

**禁止行为**：AI 完成代码修改任务后，将修改保留在工作树中即可，不得自动追加 commit/push/merge 等"收尾"操作。用户会自行决定何时提交。

## 2. 文档体系规范

### 2.1 三层记忆结构 [2026-07-20]

| 层级 | 文件 | 定位 | 更新时机 |
|------|------|------|---------|
| L1 | `Documents/global-rules.md` | 跨项目通用规范 | 用户要求同步文档时 |
| L2 | `{project}/Documents/{project}-memory.md` | 单个项目决策记忆 | 用户要求同步文档时 |
| L3 | `{project}/Documents/{project}-context.md` | 策划文档 + 技术速查 | 用户要求同步文档时 |

**注意**：三层文档的更新时机与合并 main 分支是两个独立操作。用户大多数情况下会同时说"同步文档 + 合并 main"，但理论上两者解耦——可能只同步文档不合并 main，也可能只合并 main 不同步文档。

### 2.2 上下文文档同步规则 [2026-07-16, 2026-07-20修正]

- 同步时机：由用户主动触发（说"同步文档"或类似指令），与合并 main 分支独立
- 日常开发修改时不需要同步，等用户认为一个阶段完成时再要求同步
- 同步时必须进行全覆盖检查，不能只更新与当天修改相关的几个点：
  - 操作键/按键表
  - 核心属性/玩家属性
  - 渲染常量（确认值是当前实际值）
  - 架构概览（确认新增系统模块已加入）
  - TILE系统/楼层系统等核心系统描述
  - 章节编号（无重复无遗漏）
  - 函数索引（补充所有新增函数）
  - 数据流（补充启动/探索/楼层等完整流程）
  - 游戏流程（补充探索模式+战斗模式完整流程）
  - UI面板（补充楼层信息栏等新增UI）
  - 文件清单（补充新配置文件）
  - 下一步方向（移除已完成的）
  - 过期内容清理（已淘汰的描述）

### 2.3 同步文档 + 合并 main 完整流程 [2026-07-16 优化, 2026-07-20 解耦]

> **说明**：以下描述的是用户同时要求"同步项目文档，提交并推送，并合并到main分支"时的完整操作流程。**同步文档**和**合并 main** 本质上是两个独立操作，此处放在一起描述仅因为用户通常同时要求两者。

当用户同时要求同步文档和合并 main 时：

1. **同步文档**：先更新三份文档（global-rules.md / memory.md / context.md 中涉及的部分），在"最近更新记录"顶部新增一条
2. **全面检查变更**：`git status` + `git diff` 仔细检查所有变更文件
3. **一次性提交**：所有相关变更（代码 + 文档）纳入同一个 commit
4. **推送到远程**：push 当前分支到 origin
5. **合并到 main**：切到 main → pull → merge 当前分支 → push main
6. **切回开发分支**：回到当前功能分支

**关键优化**：先更新文档再提交，避免之前"先提交代码 → 合并main → 再提交文档 → 再合并main"的重复操作。

### 2.4 多端开发记忆同步 [2026-07-20]

- **切换电脑前**：`commit + push` 当前分支（即使不合并 main），确保三份记忆文件已更新
- **切换电脑后**：`pull` 最新，然后主动告知 AI 读取三份文档
- **新会话启动**：主动提示 AI 读取 `global-rules.md` + 当前项目的 `{project}-memory.md` + `{project}-context.md`

### 2.5 global-rules 修改规则 [2026-07-20]

- **可以修改已有规定**：随着开发习惯演变，可以更新旧的规范条目
- **冲突必须询问**：如果新规范与旧规范冲突，AI 必须先向用户明确指出冲突内容，询问保留哪种规范，得到确认后才能修改
- **唯一性要求**：global-rules 内部不能有自相矛盾的内容，每一条规范必须是唯一且无歧义的
- **时间标注**：修改已有条目时，在标题日期字段追加新日期，如 `[2026-07-16, 2026-07-20修正]`

### 2.6 memory 修改规则 [2026-07-20]

- **不可修改或删除已有记录**：一旦写入的历史内容永久保留
- **新方案追加方式**：当方案从 A 改为 B 时，不能只保留 B 删除 A。必须在对应章节顶部追加 B 的记录并标注"[当前方案]"，A 的原记录保留并标注"[已废弃]"
- **反复变更完整保留**：即使出现 X→Y→X 的反复，每一步都要保留，形成完整的演变时间线
- **当前结论标注**：每个主题下必须有明确的当前结论标注（"[当前方案]"），让读者一眼看到最终状态

### 2.7 一致性原则 [2026-07-20]

| 文档 | 是否允许冲突 | 说明 |
|------|:--:|------|
| `global-rules.md` | ❌ 不允许 | 必须是唯一、无歧义的规范。新旧冲突时必须询问用户选择。 |
| CodeBuddy Memories | ❌ 不允许 | 与 global-rules.md 保持一致，两者可互相校验。 |
| `{project}-memory.md` | ✅ 允许 | 可记录冲突的决策过程（A→B→A），但必须有明确的"[当前方案]"标注。 |

### 2.8 文档写入前验证流程 [2026-07-21]

**核心原则：在对任何文档（memory.md、context.md 等）进行追加写入前，必须先完成排序验证。**

每次追加前触发以下检查：
1. **先读取全文**：确认当前最后一个已记录条目是什么日期、什么内容
2. **确认追加位置**：新条目追加在文件末尾（或同日期段落末尾），不得插入到已有条目之间
3. **同一天多个条目**：按实际对话先后顺序从上到下排列，不在条目标题中使用分钟级时间标注
4. **写入后复核**：如果涉及条目移动或重组，写入后重新读取全文确认排序正确

此规范适用于所有项目的 memory.md 和 context.md 文件。

### 2.9 规则传播检查 [2026-07-21]

**核心原则：当用户发出"同步项目文档 + 提交推送 + 合并 main"指令时，在同步三份文档的过程中，必须检查本期项目级文档（memory.md / context.md）中的新增规则是否跨项目通用，是则同步到 `global-rules.md`。**

触发时机精确绑定在用户的同步指令上，不在日常开发的每次文档修改时触发。这样做的好处：
- 同步文档本身就是一个系统性审查节点，在此处追加规则传播检查最自然、最少遗漏
- 日常开发中文档修改频繁，每次都触发反而导致麻木，不如集中在审查节点一次性处理

检查标准：
- 本期在 memory.md / context.md 中新增或修改了哪些规则/规范？
- 这些规则是否跨项目通用？（match3-rpg、tarot-battle、Project-Issac-turnbase 都适用？）
- 如果是 → 必须同步写入 `global-rules.md` 对应章节
- 如果仅适用当前项目 → 留在项目级文档即可

此检查应在 §2.3 流程的"同步文档"步骤中优先执行，与其他文档更新一同纳入 commit。

### 2.10 Memory 记录完整性要求 [2026-07-23]

**核心原则：写入 memory.md 的条目必须覆盖当天讨论的全部决策深度，不得只记录最终的代码/配置变更结果，而遗漏讨论过程中的设计原则、方法论、审计过程和策略决策。**

具体要求：
- **不只记结果**：除了最终的代码或配置变更，还要记录"为什么这样做"的推理链、被否决的替代方案、确立的分类标准等
- **审计/审查类讨论**：如果当天进行了系统性审计（如配置审查、代码检查），必须记录审计范围、分类标准、审计结论（哪类改/哪类不改/各自原因）
- **策略/流程类讨论**：如果当天确立了新的同步策略、维护流程、验证方法，必须完整记录
- **零遗漏检查**：写入 memory 前必须回溯当天完整对话，逐条确认是否有未被记录的决策或讨论内容

此规范适用于所有项目的 memory.md 文件。

## 4. AI 会话规范

### 4.1 新会话启动约定 [2026-07-20]

- AI 会自动加载 CodeBuddy Memories 中的精简规则
- 用户主动告诉 AI 读取三份文档（global-rules + memory + context）
- 三文件通过 Git 多端同步，保证不同电脑间记忆一致

### 4.2 记忆更新原则 [2026-07-20]

- **L2 memory.md** 记录标准："如果在另一台电脑不知道这件事，会不会做出错误决策或重复踩坑？" → 会就记
- 每条决策带时间戳、原因、被否决方案
- 反复变更（X→Y→X）要保留完整演变路径
- 不确定是否该记时，主动询问用户

### 4.3 不可回溯的边界 [2026-07-20]

- 如果在某台电脑上的对话没有被及时整理成 memory.md 并 push，切换电脑后无法回溯
- 解决方案：严格执行第 2.4 节的切换前 commit+push 流程

### 4.4 AI 数据真实性规范 [2026-07-21]

**核心原则：AI 不得编造任何没有数据支撑的信息。**

禁止编造的内容包括但不限于：
- **时间**：具体到分钟的时间（如"约 14:15""约 15:30"）。AI 只能感知对话先后顺序，无法获知精确时间。日内时间仅可用"上午/下午/晚间"三个粗粒度标签，或完全不标注时间仅靠条目排列顺序体现先后关系
- **数值**：未在对话中确认的数据值、参数、百分比
- **事件**：未实际发生的操作、未做出的决策、"被假设"的结论
- **状态**：未经用户确认的"已完成""已修复"等状态描述

**处理方式**：不确定的信息宁可省略或用模糊描述，绝不填充。如果某项信息确实需要但无法获取，应向用户询问而非自行推测。

### 4.5 疑问句与指令的区分 [2026-07-23]

**核心原则：用户用疑问句征求意见时，AI 只能分析和建议，不得擅自执行任何有副作用的操作。**

需要区分的话术模式：
- **疑问/征求意见**（只答不做）："你觉得需要…吗？""要不要…？""应不应该…？""有没有必要…？"
- **明确指令**（可以执行）："记录""补充""修改""执行""按照你的建议执行"

此规则覆盖以下有副作用的操作：
- 文件写入（编辑、新增、删除、重命名）
- Git 写入操作（已有 §1.4 约束，本节强化交互层面的判断）
- 命令执行
- 任何会改变项目状态的操作

**正确做法**：收到疑问句时，分析利弊 + 给出建议方案 + 等待用户明确指令后再动手。绝不在用户确认前抢先执行。
